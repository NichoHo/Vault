# Vault Phase 0 + Phase 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Phase 0 — one command (`docker compose up`) runs the whole skeleton with CI, seed, Terraform stub. Phase 1 — a working OIDC round trip: the Next.js storefront signs in via the self-built Go IdP (register/login, sessions, Auth Code + PKCE, RS256 JWT + JWKS, consent, audit log) plus listings CRUD in the market service.

**Architecture:** Two Go services (`id` :8081, `market` :8082) on one Postgres (schema-per-service), Next.js storefront (:3000) that proxies `/idp/*` to the id service so IdP cookies are same-origin in dev. No gateway, no gRPC, no outbox, no MFA, no refresh tokens — those are Phase 2/3 per the spec.

**Tech Stack:** Go 1.26 (stdlib `net/http` ServeMux — Go ≥1.22 method routing; no chi needed yet), `jackc/pgx/v5`, `golang.org/x/crypto/argon2`, hand-rolled RS256 JWT (the "built from RFCs" brand), Next.js 15 App Router + TypeScript + Tailwind v4, Postgres 17, Docker Compose, GitHub Actions.

## Global Constraints

- Single Go module `vault` at repo root; services under `cmd/`, logic under `internal/`.
- Postgres: one database `vault`, schemas `id` and `market`; each service migrates its own schema at startup via a tiny embedded runner (no golang-migrate).
- Money: `int64` minor units, currency JPY (single currency per spec).
- Passwords: argon2id, PHC string format (`$argon2id$v=19$m=65536,t=1,p=4$...`).
- Tokens: RS256 access + id tokens, 15-min expiry, `iss` = `ID_ISSUER` env (default `http://localhost:8081`), `aud` = `vault`. Auth codes: 5-min expiry, single-use.
- OAuth client `vault-web`: public client, PKCE S256 **required**, redirect `http://localhost:3000/auth/callback`.
- Design tokens (spec §6): ink `#16161A`, paper `#FAF8F5`, surface `#FFFFFF`, torii `#D9381E`, moss `#2E7D5B`, kohaku `#C98A0B`, indigo `#31456A`; Inter + Noto Sans JP via `next/font`; `tabular-nums` on money. No dark mode.
- DB-backed Go tests read `TEST_DATABASE_URL` and `t.Skip` when unset; CI provides a Postgres service and runs `go test -race`.
- Local Go on Windows: user-space install from official go.dev zip (no admin); race detector runs in CI (Linux), not locally.

### Task 5: id — RS256 JWT + JWKS (Phase 1)

**Files:** Create `internal/id/jwt.go`, `internal/id/jwt_test.go`.

**Interfaces:**
```go
type Signer struct { Kid string; Key *rsa.PrivateKey }
func LoadOrCreateSigner(ctx, pool) (*Signer, error) // active row in signing_keys, else generate 2048-bit + insert
type Claims struct { Iss, Sub, Aud string; Exp, Iat int64; Email, Nonce string `json:",omitempty"` }
func (s *Signer) Sign(c Claims) (string, error)
func VerifyJWT(token string, keys map[string]*rsa.PublicKey, iss, aud string, now time.Time) (Claims, error)
func JWKS(keys map[string]*rsa.PublicKey) ([]byte, error) // {"keys":[{kty,alg,use,kid,n,e}]}
```
`VerifyJWT` errors on: bad structure, unknown kid, alg != RS256 (reject `none`), bad signature, exp < now, iss/aud mismatch.

**Tests:** sign→verify roundtrip; tampered payload fails; expired fails; wrong aud fails; `alg:none` rejected; JWKS output parses and n/e reconstruct the public key.
- [ ] Commit: `feat(id): hand-rolled RS256 JWT + JWKS`

### Task 7: id — authorize / consent / token (the OIDC core) (Phase 1)

**Files:** Create `internal/id/oauth.go`, `internal/id/oauth_test.go`; extend `server.go` routes.

**Flow:**
- `GET /authorize` (params `response_type=code`, `client_id`, `redirect_uri`, `scope`, `state`, `code_challenge`, `code_challenge_method=S256`, optional `nonce`): validate client + exact redirect_uri match + S256 required. No session → 302 `WEB_URL/auth/login?return_to=<urlencoded original>`. No consent row → 302 `WEB_URL/auth/consent?client_id=&scope=&return_to=`. Else insert auth_code (5 min) → 302 `redirect_uri?code=&state=`. Audit `code.issue`.
- `POST /consent {client_id, scope}` (session) → upsert consents, 204. Audit `consent.grant`.
- `POST /token` (form: `grant_type=authorization_code`, `code`, `redirect_uri`, `client_id`, `code_verifier`): single query marks code used (`UPDATE ... SET used_at=now() WHERE code=$1 AND used_at IS NULL RETURNING ...`) so replay fails atomically; check expiry, client, redirect_uri; PKCE check `base64url(sha256(verifier)) == code_challenge`; on success → `{access_token, id_token, token_type:"Bearer", expires_in:900}`; failures → 400 `{"error":"invalid_grant"}`. Audit `token.issue`.
- `GET /.well-known/openid-configuration`, `GET /.well-known/jwks.json`.

**Misuse tests (spec §9, the differentiator — full round trip via httptest against real DB):**
```go
// happy path: register → authorize(session cookie) → consent → authorize → code → token → VerifyJWT ok
// wrong code_verifier            → 400 invalid_grant
// replayed auth code             → 400 invalid_grant (second exchange fails)
// expired code (insert expires_at=-1m directly) → 400 invalid_grant
// redirect_uri mismatch at /authorize → 400, no redirect
```
- [ ] Commit: `feat(id): auth code + PKCE, consent, token, JWKS discovery`
